Static vs. Dynamic Typed Language

Imagine

You have access to a very intricate machine, something from the pages of one of Dr. Suess' wispy stories. This device will take in materials and build items of your choosing while you watch. Now the first item you desire the machine to build is a fabulous, glorious house.

Here are 2 lists of building materials:

List 1

Concrete, rebar, brackets, wood, plastic, glass, drywall,... (you get the idea)

List 2

Graham crackers, frosting, candy canes, gumdrops, small marshmallows, licorice whips, sprinkles, peppermint candies,

red hots,... (you get the idea)

At the moment, the machine is set to check the input materials very carefully and decide if they are suitable for the item's construction. It has preprogrammed parameters regarding the types of materials and their purposes. It has been instructed regarding certain specifications for the load bearing beams, which will hold up the ceiling, the bulk modulus for the concrete which will form the foundation, the size of nails to be used in affixing one piece of wood to another, and so on.

With these parameter, the machine will build a very nice house. If it is given some creative license, it will design houses with different architectural styles and this will be very pleasing. However, resulting from its current settings, it will never utilize glass as a load bearing beam.

You might ask why?

The machine has been told, in advance, that load bearing beams can only be made from one, two, or three different material choices - glass is not on this list. This has its pros and cons. All structures it designs and builds will be safe and adhere to certain codes. However, this might limit it creativity. For example, it will never build you a house made from List 2... well, unless it has been told to make a Gingerbread House and use items from List 2. In this case it has a new set of criteria upon which to navigate decisions and manipulate information.

Static Typed Languages

These languages, like our machine above on its current settings, specify ahead of time what type of data is allowed to be assigned to certain variables and then very importantly, what type of manipulations, operations, or interactions may happen to or with this data. This defining is embedded in very fabric of the code. This seems nice and in many ways it is. It is like having very strict air traffic control and anyone who travels by flight is happy that someone is guiding all of the planes to their proper destinations (Most of us would rather not think about this aspect while in flight).

This level of control reduces runtime and guarantees types, which is useful for the certainty of return values. However, the code is not as reusable. Remember, it was written with one very specific goal/task in mind. Hopefully it carries out this goal/task with efficiency, but this removes quite a bit of its flexibility, much like the house example above.

An example of some static typed language that should work well:

fun cube x = x * x * x

cube 3

This code cubes a number. People should know better than to expect it to do anything other than take in numbers, not arrays, nor strings. These enforced rules sometimes create situations where the logic of the program is sound, but the program is rejected because it does not follow all of the data type rules.

Here is a favorite that's also presented in the following youtube video: (15. Static Versus Dynamic Typing - Part 1)

var par_of_pars = f (fn x => (x,x))*

*I ran into this problem when I wanted to create a pair out of just one input. You can create a pair out of just one input, but not the way just mentioned. The example makes sense, but it does not follow the rules and will not run as written.

Dynamic Typed Languages

When looking at dynamic typed languages, many of the restrictions of data types are lifted. It is the programmers responsibility to put together a program that uses information correctly, since there is less intrinsic guarantee that the language is being guided and governed. It is believed that this increases creativity and productivity. With dynamic typed language you would, on occasion, get a house made of the items from list 2. Depending on the size house desired and the mood you are in, this could be tragic or magnificent.

This code is more general, remember data types have not been explicitly given. Since data type defining is not as clear, much of the enforcing takes places during running. This does increase running time, which is a con. Much of the beauty comes from variables being defined more by their qualities and functionality at a given State than their initial values. This allows for information to be more fluid and to change over time.
